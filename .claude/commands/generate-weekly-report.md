# LIFEPLUS 주간 리포트 생성

이 스킬을 실행하면 `data_MMDD.json`을 기반으로 주간 리포트 HTML을 생성한다.

---

## 0단계 — 데이터 파일 확인

현재 작업 디렉토리에서 가장 최근 `data_MMDD.json` 파일을 찾아 읽는다.

`channel_kpi` 섹션의 값이 0이거나 기본값(`"W__"`, `"YYYY.MM.DD"`)이면, 사용자에게 아래 항목을 요청한다:

- 이번 주 라벨 (예: W26)
- 집계 기간 (예: 2026.06.21–06.27)
- 채널 전체 구독자 수 (누적)
- 이번 주 구독자 증감
- 채널 전체 주간 조회수
- 연간 누적 조회수
- HIT 달성 에피소드 수 (누적)

---

## 1단계 — 주간 리포트 생성

**출력 파일명**: `LIFEPLUS_weekly_report_WXX_N.html` (예: `W27_1`)

가장 최근에 생성된 `LIFEPLUS_weekly_report_WXX_N.html`(git 히스토리 포함, `git log --diff-filter=A -- "LIFEPLUS_weekly_report_*.html"`로 탐색)을 템플릿으로 활용한다. 로컬에 파일이 없으면 git 커밋에서 가져온다.

### 반드시 갱신할 항목

1. **헤더 KPI**:
   - 구독자 증감: `channel_kpi.subscribers_weekly`
   - 주간 조회수: `channel_kpi.views_weekly_total`
   - 발행 편수: 이번 주 발행 에피소드 수
   - 평균 CTR: 이번 주 발행 에피소드들의 CTR 평균
   - 누적 조회수: `channel_kpi.views_ytd_total`

2. **연간 목표 달성현황**:
   - 구독자: `subscribers_total / 1,100,000`
   - 조회수: `views_ytd_total / 60,000,000`
   - HIT: `hit_count / 12`

3. **이번 주 발행 에피소드**: JSON `pending`(status=collecting) + 최근 7일 내 `episodes`
   - 1D 집계 중인 에피소드: 1D 지표 사용
   - 7D 집계 완료된 에피소드: 7D 지표 사용

4. **IP별 성과 비교**: IP(위클리/H&N/딥다이브)별 1D·7D·28D 최근 2편씩, 총 9개 표
   - **컬럼은 1D·7D·28D 모두 동일하게 10개 통일**: 에피소드·발행일·조회수·자연유입%·평균조회율·구독↑·CTR·시청시간·반응·등급 (28D도 `vr_28d`·`eng_28d` 필드를 채워 1D/7D와 동일 구조로 표시 — 컬럼 개수를 줄이지 않는다)
   - `eng_1d`(반응 합계: 좋아요+공유+댓글)는 엑셀 39번 컬럼에 항상 존재하는 값 — `status=collecting` 편에서 이 값이 JSON에 없다면 데이터 자체가 없는 게 아니라 `extract_lifeplus_data.ps1` 추출 버그일 가능성이 높으니, 값을 추정/역산하지 말고 먼저 스크립트의 collecting 분기(`if ($v1d) {...}` 블록)에 `$ep.eng_1d = $eng1d` 대입이 빠졌는지 확인하고 고친 뒤 재추출할 것. 그래도 정말 없다면 `-` + 각주 처리
   - 각 IP의 1D/7D/28D 최근 2편은 매주 데이터 갱신에 따라 자동으로 밀려남(예: 이번 주 새로 7D 확정된 에피소드가 다음 주엔 28D 최근 2편 후보가 됨) — 매주 전체 시즌 데이터를 다시 스캔해서 "최근 2편"을 재선정할 것

5. **BEST/WORST 분석**:
   - 원칙: `자연유입%`(nat_pct) 기준 IP 내 전체 시즌 상/하위 랭킹 (조회수 절대값이 아님 — 팬덤성 소재는 조회수는 높아도 자연유입이 낮을 수 있음)
   - 이번 주 새로 7D 확정된 에피소드가 상/하위권에 신규 진입하면 "(신규)" 배지를 달고, 진입하지 않으면 배지 없이 기존 순위 유지
   - 매주 전체 시즌 데이터를 재스캔해서 순위를 갱신 — 지난 주 픽을 그대로 복사하지 말 것

6. **다음 주 발행 예정**: `pending` 중 status=`pending`인 에피소드 (IP별 최초 1편, 발행 요일 주기는 직전 발행일 기준으로 추정)

## 2단계 — 검증 체크리스트

- [ ] 헤더 구독자 증감 = `channel_kpi.subscribers_weekly`
- [ ] 연간 달성률 % 계산 정확성 (구독자·조회수·HIT 각각)
- [ ] 시청시간 표기 = `watch_h × 60 / views` (소수점 1자리)
- [ ] 이번 주 에피소드 수 = 헤더 발행 편수와 일치
- [ ] IP별 성과 비교 1D·7D·28D 표 3개 모두 컬럼 10개(평균조회율·반응 포함)로 동일한지 확인
- [ ] 신규 1D 편의 반응 값이 JSON에 없으면 먼저 `extract_lifeplus_data.ps1`의 collecting 분기에 `eng_1d` 대입 누락이 있는지 확인 — 진짜 데이터 부재일 때만 `-` + 각주
- [ ] BEST/WORST가 이번 주 신규 확정 데이터 기준으로 재스캔됐는지 (지난 주 표 그대로 복붙 아닌지)
- [ ] 지난 주 리포트에서 "집계중"으로 남겨뒀던 트렌드 표 행이 이번 주 7D 확정됐다면 실제 수치로 소급 갱신
- [ ] 브라우저에서 정상 렌더링 확인 요청

### 고친 흔적

1. **channel_kpi 기본값 감지 분기** — `"W__"` / `"YYYY.MM.DD"` 같은 기본값이 남아 있으면 사용자에게 7개 항목을 한 번만 물어보고 진행하도록 0단계에 분기 추가; 자동으로 채워져 있으면 묻지 않음
2. **1D/7D 집계 분기** — 발행 직후 에피소드가 7D 집계 기준으로 잘못 표시되던 문제 → `status=collecting`이면 1D 지표 우선 사용으로 명시
3. **28D 표 컬럼 통일** — 28D 표가 1D/7D보다 컬럼이 2개(평균조회율·반응) 적어 정보 비대칭 발생 → `vr_28d`/`eng_28d` 채워 10컬럼으로 통일
4. **`extract_lifeplus_data.ps1` collecting 분기 `eng_1d` 누락 버그 수정** — 신규 발행(`status=collecting`) 에피소드의 반응 합계가 리포트에 `-`로 나온 원인은 데이터 부재가 아니라 추출 스크립트 버그였음. 엑셀 39번 컬럼(Eng1D)을 `$eng1d` 변수로 읽어놓고도, `complete` 분기(285번 줄)에는 `eng_1d = $eng1d`가 있지만 `collecting` 분기(227~245번 줄)에는 이 대입이 누락되어 있었음 → collecting 분기에 `$ep.eng_1d = $eng1d` 추가 후 재추출로 해결. **교훈**: JSON에 값이 비어있다고 바로 "원본 데이터 없음"으로 단정하지 말고, 사용자가 "엑셀엔 있다"고 하면 먼저 추출 스크립트의 필드별 대입 누락 여부를 의심할 것

## 참고: 파일 위치

```
<프로젝트 루트>/                             ← git clone 위치 (Claude Code 작업 디렉토리)
├── 지표관리_MMDD.xlsx                        ← 원본 엑셀 (매주 업데이트)
├── extract_lifeplus_data.ps1                ← 엑셀 → JSON 변환 스크립트 (컬럼 인덱스는 $COL 참고)
├── data_MMDD.json                           ← 추출된 데이터
└── LIFEPLUS_weekly_report_WXX_N.html        ← 출력 리포트
```

리포트의 특정 필드가 비어있거나(`-`, `null`) 사용자가 "엑셀엔 값이 있다"고 지적하면, JSON을 탓하기 전에 `extract_lifeplus_data.ps1`에서 해당 필드가 실제로 읽혀서(`$ws.Cells(...)`) `$ep` 객체에 대입되는지부터 확인한다. 컬럼 인덱스 자체가 틀렸으면 `-ShowHeaders` 옵션으로 실제 헤더 위치를 재확인 후 `$COL` 값을 수정한다.
