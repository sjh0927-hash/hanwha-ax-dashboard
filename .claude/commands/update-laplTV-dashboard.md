이 스킬을 실행하면 최신 엑셀 보고서를 읽어 `라플TV_캠페인_대시보드.html`을 업데이트한다.

## 핵심 개념: 날짜 오프셋 (2026-07-06부터 규칙 변경)

**2026-07-06 이후 파일(예: `..._260705.xlsx`)부터: 오프셋 없음.**
엑셀 파일명의 날짜 = 마감일자(실제 광고 집행 마지막 날) 그 자체. `260705.xlsx` → HTML `'7/5'`까지 그대로 반영.
`update_dashboard.ps1`도 이에 맞춰 47번 줄의 `$endDay = ... - 1`에서 `- 1`을 제거함 (2026-07-06 수정).

**2026-07-05 이전 파일은 아래의 구(舊) 1일 오프셋 규칙이 적용됐었다** (과거 리포트 데이터를 다시 볼 일이 있을 때만 참고):

> 엑셀의 날짜 = 광고 데이터 보고 기준일 (= 실제 광고 집행일 + 1일)
> HTML 레이블 = 실제 광고 집행일
> 따라서: HTML '6/N' = Excel (N+1)일 행
> 예) 엑셀 6월 25일(46197) 행 → HTML `'6/24'` 레이블로 입력

**Why:** 광고 대행사/데이터 제공처가 엑셀 파일명에 넣는 날짜의 의미를 "생성일(익일)"에서 "마감일자(당일)"로 변경했다고 사용자가 확인함. 새 파일이 오면 항상 최근 파일이 신규칙, 과거 파일은 구규칙이라는 걸 헷갈리지 말 것 — 파일명만 보고 규칙을 추측하지 말고, 이 문서의 최신 날짜 이후 파일인지부터 확인.

## ⚠️ 필수 주의사항

### replace_all 사용 절대 금지

HTML 내 `var data=[{lbl:'6/XX',...}]` 패턴이 여러 차트에 동시에 존재한다.  
`replace_all`로 날짜 배열을 교체하면 **의도하지 않은 다른 차트 데이터도 함께 덮어써진다**.

예: 전환 롤링 7일 차트(`cv1`)와 클릭최대화 프로그램별 차트(`clickProg`)가 동일 패턴 공유  
→ `replace_all` 시 `clickProg` 데이터가 전환 데이터로 교체됨 → `d.c` 속성 없음 → 차트 미표시

**항상 충분한 주변 컨텍스트로 단건 교체(replace_all:false)할 것.**

### CC 객체 새 날짜 추가 시 버그 주의

CC 객체(`전환 소재별`)에 새 날짜를 추가할 때, **직전 날짜 값이 그대로 복사되는 버그**가 발생한 이력이 있다.  
새 날짜 추가 후 반드시 직전 날짜 값과 비교 검증할 것 (군체/토이스토리/스필버그 3개 소재 모두).

## 🗓️ 월 전환 시 필수 체크리스트 (매월 1회, 매우 중요)

> 2026-07 전환 시 클릭최대화·전환·후속조회 3개 탭에서 KPI·프로그램별 성과·각종 차트·소재별
> 테이블·일자별 상세보기가 동시다발적으로 틀어져 있던 걸 발견해서 정리한 것.
> **근본 원인은 하나다**: 월이 바뀌면서 새로 시작한 소재(회차)가 있었는데, 기존에 알던 소재
> 이름만 계속 갱신하고 "엑셀에 새 소재 블록이 생겼는지"를 확인하지 않아서 그 소재 데이터가
> 통째로 누락됐다. 그 결과 프로그램별 합계가 소재별 합계와 안 맞고, 여러 차트/테이블이 전달
> 값 그대로 남아있었다. **다음 달 전환 때는 아래 A~G를 순서대로 확인할 것.**

### A. 신규 소재 탐지 — 가장 먼저 할 일

"이미 아는 소재 이름 목록"만 기준으로 값을 갱신하지 말고, **엑셀 소재별 데이터 섹션의 헤더
행 전체를 매번 다시 스캔**해서 새로 생긴 소재 블록이 있는지 확인한다.

```powershell
# 디맨드젠_클릭최대화 시트 예시 — 소재별 데이터 블록은 10칸 간격(B,L,V,AF,AP,AZ...)으로 반복
$blockStarts = 2,12,22,32,42,52   # 시트/월마다 개수·위치가 달라질 수 있음
for ($r=1; $r -le $used.Rows.Count; $r++) {
  foreach ($bs in $blockStarts) {
    $nameCell = $vals[$r,$bs]
    if ($nameCell -is [string] -and $vals[$r+1,$bs] -eq '기간') {
      Write-Output "HEADER row=$r col=$bs name=$nameCell"
    }
  }
}
```

이렇게 뽑은 전체 헤더 목록을 **JUNE/CC/RD/CREATIVES 등 HTML 쪽 객체에 이미 있는 소재 이름과
대조**한다. 목록엔 있는데 HTML엔 없으면 신규 소재 → 반드시 추가.

**2026-07 실사례**: 디맨드젠_클릭최대화 시트에 뮤지컬1(20화)·뮤지컬2(21화)·더베어(HOT17화)·
뉴뮤지엄(딥7화)·장송의프리렌(딥8화) 5개 신규 소재 블록이 이미 있었는데, JUNE 객체는 스필버그·
토이스토리(13화)만 계속 갱신하고 있어서 5개 전부 누락된 채로 며칠이 지났다. 이 때문에 KPI
노출/클릭/CPC/CPV가 전부 틀어져 있었다.

### B. "합계" 행을 맹신하지 말 것 — 신규 소재 컬럼의 스프레드시트 복사 버그

신규 소재 컬럼은 기존 소재(주로 직전 회차)의 수식·서식을 복사해서 만드는 경우가 많다. 그 결과
**신규 소재가 실제로 시작하기 전 구간에는 옆 컬럼(또는 이전 회차)의 값이 그대로 찍혀 있는
경우**가 있다. 2026-07 사례에서는 뮤지컬1/뮤지컬2/토이스토리(HOT16화) 3개 컬럼이 6/23~6/30
구간에 완전히 동일한 값을 보였다 — 실제로는 뮤지컬1/2가 그 기간에 존재하지 않았는데 옆 컬럼
(토이스토리 HOT16화)의 실제 값이 그대로 복사되어 있었던 것. 심지어 "합계" 라벨이 붙은 요약
행 하나도 3개 컬럼 모두 동일한 값을 보여줘서, 요약 행만 보고는 이 버그를 알아챌 수 없었다.

**대응**: 신규 소재가 시작한 것으로 보이는 날짜 이전 구간에서 인접한 여러 소재 컬럼 값이
완전히 똑같다면 그 구간은 버리고, **다른 컬럼과 값이 달라지기 시작하는 첫 날**부터만 진짜
데이터로 취급한다. 반드시 raw 셀을 여러 개 나란히 찍어서 눈으로 비교해 판단할 것.

### C. 요일 계산은 항상 `new Date(2026, month-1, day).getDay()` — `day % 7` 금지

`day % 7` 같은 하드코딩 패턴은 "이번 달 1일이 무슨 요일인가"에 따라 옵셋이 달마다 달라지므로
월이 바뀌면 반드시 틀린다. 2026-07 전환 때 클릭최대화/전환/후속조회 3개 탭의 "일자별 OOO
현황" 섹션과 개요 탭 `cv1` 차트에서 전부 이 패턴이 쓰이고 있었고, 7월 데이터를 추가하자마자
요일이 실제 달력과 어긋나는 버그가 반복 발생했다.

```javascript
// 나쁜 예 (월 경계에서 깨짐)
var dowStr = ['일','월','화','수','목','금','토'][day % 7];

// 좋은 예 (항상 정확 — 2026년 기준, month는 1~12)
var dowStr = ['일','월','화','수','목','금','토'][new Date(2026, month-1, day).getDay()];
```

### D. "일자별 OOO 현황" 3개 탭 공통 패턴 — 월 구분 라벨 + 신규 월 버튼 추가

클릭최대화(`click-day-btns`)·전환(`conv-day-btns`)·후속조회(`retarget-day-btns`) 3개 탭 모두
구조가 동일하다. 월이 바뀔 때마다 아래를 그대로 적용한다:

1. 버튼 목록 앞에 이전 달 라벨(`6월`)을 넣고 이전 달 버튼(1~30/31)을 렌더링
2. 그 뒤에 새 달 라벨(`7월`)을 넣고 새 달 버튼(1~보고서기준일)을 렌더링
3. `data-date`/`data-cdate`/`data-rdate` 값은 `"7/N"` 형식으로 새 달 버튼에만 부여
4. 기본 선택(`on` 클래스)과 초기 렌더 호출(`renderDateCards('7/N')` 등)을 **최신 날짜**로 변경
5. 렌더 함수 내부에서 `key.split('/')`로 **월도 함께 파싱**해서 노트 텍스트의 "N월"과 요일
   계산(C번 항목)에 반영 — 이걸 빠뜨리면 `mo is not defined` 런타임 에러가 난다
6. 섹션 타이틀(`.sect-lbl`)도 "· 7월" → "· 6월~7월"로 바꿔 두 달치가 섞여 있음을 표시

```javascript
// 버튼 렌더링 템플릿 (세 탭 공통, 신규월마지막일 = 보고서 기준일)
var h='<span style="font-size:11px;color:var(--faint);align-self:center;letter-spacing:1px;margin-right:2px;">6월</span>';
for(var d2=1;d2<=30;d2++){
  var isWk=['일','토'].indexOf(['일','월','화','수','목','금','토'][new Date(2026,5,d2).getDay()])>=0;
  h+='<button class="date-btn'+(isWk?' wk':'')+'" data-date="6/'+d2+'">'+d2+'</button>';
}
h+='<span style="font-size:11px;color:var(--faint);align-self:center;letter-spacing:1px;margin:0 2px 0 10px;">7월</span>';
for(var d3=1;d3<=신규월마지막일;d3++){
  var isWk3=['일','토'].indexOf(['일','월','화','수','목','금','토'][new Date(2026,6,d3).getDay()])>=0;
  var cls3='date-btn'+(isWk3?' wk':'')+(d3===신규월마지막일?' on':'');
  h+='<button class="'+cls3+'" data-date="7/'+d3+'">'+d3+'</button>';
}
```

### E. 차트별 "당월만" vs "월별 누적 유지" 규칙 (2026-07 확정)

같은 "일별 추이" 류 차트라도 성격이 다르다. 2026-07 전환 때 사용자가 명시적으로 확정한 규칙:

| 차트 | 규칙 | 이유 |
|------|------|------|
| `cv1` (개요, 일별 구독전환수) | **당월만** | 7일 롤링 차트라 다음 달이면 어차피 새로 채워짐 |
| `convDaily` (전환, 일별 구독전환수&CVR) | **당월만** | 사용자가 "6월 제외, 7월 실적만" 명시적으로 요청 |
| `retargetDaily` (후속조회, 일별 조회수&CTR) | **당월만** | 위와 동일 + 미집행이면 0으로 표시해 미집행 상태를 그대로 보여줌 |
| `clickTrend` (클릭최대화, 월별 CTR 추이) | **월별 누적 유지, 새 달 추가만** | 제목 자체가 "월별" 추이 — 여러 달 비교가 목적 |
| "일자별 OOO 현황" 버튼+카드 (3개 탭) | **이전 달 + 당월 모두, 월 구분 라벨로 표시** | 일 단위 상세 조회 기능 — 과거 날짜도 계속 조회 가능해야 함 |

표에 없는 새 케이스면 차트 제목에 기간이 괄호로 명시돼 있는지 먼저 본다 (`(7/1~7/5)`처럼
써 있으면 그 기간만 표시하는 게 기본값). 애매하면 사용자에게 먼저 확인할 것.

### F. 프로그램별 합계 = 소재별 합계 교차검증 (필수)

KPI 박스·프로그램별 성과 카드 숫자는 반드시 **소재별로 갱신한 데이터를 프로그램 단위로
합산한 값과 일치**해야 한다. 엑셀에 프로그램별 합계 표가 이미 있으면(예: 디맨드젠_클릭최대화
시트 맨 위 **"1. SUMMARY"** 표 — R104 소재별 합계를 일일이 더할 필요 없이 프로그램별 광고비·
노출·조회·클릭·CTR·CPV가 바로 나와 있다) 그 표 값을 그대로 쓰되, HTML에 반영한 소재별
데이터의 합과 반드시 대조한다.

```bash
node -e "
const fs = require('fs');
const html = fs.readFileSync('라플TV_캠페인_대시보드.html', 'utf8');
const m = html.match(/var JUNE=\{([\s\S]*?)\n\};/);
const JUNE = eval('({' + m[1] + '})');
let progCost={weekly:0,hot:0,deep:0};
['7/1','7/2','7/3','7/4','7/5'].forEach(k=>{
  JUNE[k].forEach(c=>{ progCost[c.prog]+=c.cost; });
});
console.log(progCost);  // 엑셀 '1. SUMMARY' 표의 프로그램별 광고비와 비교
"
```

이 방식으로 2026-07 사례의 모든 숫자(라플위클리 669,757원, HOT&NEW 1,433,880원, 딥다이브
289,285원 등)를 소수점 오차 수준까지 검증했다. **주의**: 처음 clickProg 차트를 고칠 때 JUNE
객체가 아직 불완전한 상태에서(A번 버그 발견 전) 계산해서, 딥다이브 몫을 HOT&NEW로 잘못
배정한 채 한 번 배포한 적이 있다 — 반드시 A번(신규 소재 탐지)을 먼저 끝내고 나서 이 교차검증을
해야 숫자가 맞는다.

### G. 캠페인이 "미집행" 상태일 때 표기 컨벤션

특정 캠페인/프로그램이 해당 월에 전혀 집행되지 않았으면 (예: 2026-07 후속조회 캠페인):

- KPI/프로그램카드 배지: `<span class="badge" style="background:rgba(255,71,87,.12);color:var(--red);">미집행</span>`
- 카드 전체에 `opacity:.6`을 추가해 시각적으로 비활성 느낌을 준다
- 수치는 `0`, 비율(CTR/CVR 등)은 `-`로 표기 (0%로 쓰면 "집행은 했는데 성과가 0%"처럼 오해될 수 있음)
- 소재별 테이블에서 직전까지 활성이던 소재는 "활성"→"중단"으로 상태값을 바꾸고 운영기간에
  종료일(보통 전월 말일)을 명시, 해당 월 실적 컬럼은 0/`-`로 정리
- "일별 추이" 차트는 E번 규칙대로 당월 구간만 만들고 값은 전부 0으로 — 차트 자체를 숨기지
  않는다 (0이 찍힌 빈 차트가 "미집행"이라는 사실을 가장 명확하게 보여준다)

## 0단계 — 파일 확인

**최신 엑셀 보고서**: 아래 PowerShell로 자동 감지한다.

```powershell
$projRoot = (Get-Location).Path
```

**HTML 위치**: `id="page-click"` → `<canvas id="clickDonutCost">`, `<canvas id="clickDonutClicks">`  
**JS 위치**: `drawDonut('clickDonutCost', [...])`, `drawDonut('clickDonutClicks', [...])`

**데이터 소스**: `$wsClick` 소재별 6월합계 행(R104) — 소재별 비용/클릭수

> ⚠️ **중앙 수치 하드코딩 금지** — `drawDonut(canvasId, slices, centerSub, fmt)`는 중앙에 표시되는 총계(예: `945만원`, `23.9만`)를 **slices 배열의 `v` 합계에서 자동 계산**한다 (`fmt` 콜백으로 단위만 지정).  
> 과거 이 값을 문자열로 직접 써넣다가(`'945만원'`처럼) 소재별 `v` 값은 갱신하고 중앙 텍스트는 깜빡 잊는 일이 반복돼서, 도넛 조각과 중앙 숫자가 서로 다른 값을 보여주는 버그가 여러 차례 재발했다 (커밋 `f6470f1`은 이 증상을 슬라이스 데이터 수정 없이 라벨 문자열만 바꿔 임시 봉합한 사례).  
> 지금 구조에서는 **소재별 `v` 값만 갱신하면 중앙 수치는 자동으로 맞다** — 별도로 손댈 부분 없음. 새 소재 추가/종료 시에도 슬라이스 배열만 수정하면 된다.

**HOT&NEW_조회** (`HOT & NEW_조회`):

- 회차별 6월합계 읽기

### HOT&NEW 16화 특별 처리

HOT&NEW 16화는 **조회캠페인 + 클릭최대화 캠페인** 합산값을 CREATIVES에 표시:
- `imp` = 조회시트 imp + `$wsClick.Cells(104,25)` (클릭최대화 6월합계)
- `views` = 조회시트 조회수 + `$wsClick.Cells(104,26)`
- `clicks` = 조회시트 클릭수 + `$wsClick.Cells(104,27)`
- `ctr` = total_clicks / total_imp * 100
- `vtr` = total_views / total_imp * 100
- `cpv` = total_cost / total_views (반올림)

