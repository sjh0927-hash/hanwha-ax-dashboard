# LIFEPLUS TV Score 리포트 생성

이 스킬을 실행하면 `data_MMDD.json`을 기반으로 TV Score 리포트 HTML을 생성한다.

---

## 0단계 — 데이터 파일 확인

현재 작업 디렉토리에서 가장 최근 `data_MMDD.json` 파일을 찾아 읽는다.

파일명의 MMDD (예: `data_0624.json` → `0624`)를 출력 파일명에 사용한다.

---

## 1단계 — LTV Score 계산

### IP별 벤치마크 (기대값) — 실제 구현 기준 (0706 검증 완료)

**자연유입은 두 가지 기준을 가중 절충한다** (`N(nat_pct)×0.1 + N(nat_abs)×0.9`, 2026-07-14부터 — 과거엔 5:5 단순평균) — nat_pct 단독은 광고비 많은 영상에 불리, nat_abs 단독은 신규/저조회 영상에 불리하기 때문에 절충 자체는 유지하되, **nat_pct_7d + ad_pct_7d = 1(항등식, 상관계수 -1.000 실측 확인)**이라 자연유입율이 콘텐츠 품질이 아니라 광고 집행 여부의 거울상이라는 게 밝혀져 비중을 1:9로 낮춤(라플위클리 누적 프로젝트 `calc_ltv.js`에서 먼저 검증 후 이 리포트에도 동일 적용). 아래 표는 실제 HTML `BENCH_W/H/D` 상수와 일치하도록 검증됨 (`LIFEPLUS_TV_Score_0629_Report.html` DATA 배열 역산으로 마이클/아르테미스2호/성난사람들2/골드랜드 4편 검증).

| IP | nat_pct (자연유입%) | nat_abs (자연유입 조회수) | vr (조회율) | ctr | eng (참여도, 절대건수) | sub (구독자증감) |
|----|-----|-----|-----|-----|-----|-----|
| 위클리S6 (w) | 0.317 | 100,196 | 0.184 | 0.0364 | 2,500 | 180 |
| HOT&NEW (h) | 0.184 | 27,286 | 0.209 | 0.0379 | 750 | 84 |
| 딥다이브 (d) | 0.209 | 54,618 | 0.171 | 0.0330 | 1,400 | 97 |

> ⚠️ 과거 버전 문서는 `eff (반응효율)` 항목을 0.0074/0.0050/0.0057 같은 비율값으로 표기했으나, 실제 구현은 `eng_7d`(좋아요+공유+댓글 절대수)를 2,500/750/1,400 벤치마크로 정규화한다. 다음에 이 표를 볼 때 실제 HTML의 `BENCH_W/H/D` 상수와 다르면 이 표가 아니라 HTML을 신뢰할 것.

### 정규화 함수

```
N(v, bench) = min(1.0, max(0, v / (bench × 2)))
```

### LTV Score 공식

```
N_nat = N(nat_pct_7d, IP.nat_pct)×0.1 + N(nat_abs_7d, IP.nat_abs)×0.9   ← 가중 절충 공식, 필수 (2026-07-14부터 1:9, 과거 5:5)

Base = N_nat        × 0.30
     + N(vr_7d)      × 0.25
     + N(ctr_7d)     × 0.20
     + N(eng_7d)     × 0.15   (eng_7d 없으면 0으로 처리)
     + N(sub_7d)     × 0.10

v2 = Base + b_org + b_algo + b_srch - pen
```

### 보너스/패널티 기준

| 항목 | 조건 | 값 |
|------|------|-----|
| b_org (자연유입 우수) | nat_pct_7d ≥ 0.30 | +0.05 |
| b_algo (알고리즘 유입) | min(0.05, algo_pct_7d × 0.5) | 0~0.05 |
| b_srch (검색 유입) | min(0.03, srch_pct_7d × 0.6) | 0~0.03 |
| pen (복합 패널티) | vr_7d < 0.10 → −0.025 / sub_7d < 0 → −0.025 / ctr_7d < 0.02 → −0.025 (조건마다 누적) | 최대 −0.05 |

> eng_7d가 JSON에 없으면 N(eng)=0으로 처리. (구 버전 문서의 `*eff미반영` 주석 규칙은 폐기 — eng_7d는 통상 항상 존재하며 정상적으로 점수에 반영된다.)

### 등급 기준

| 등급 | 조건 |
|------|------|
| S | v2 ≥ 0.70 |
| A | 0.50 ≤ v2 < 0.70 |
| B | 0.35 ≤ v2 < 0.50 |
| C | v2 < 0.35 |

## 2단계 — TV Score 리포트 생성

**출력 파일명**: `LIFEPLUS_TV_Score_MMDD_Report.html`

가장 최근에 생성된 `LIFEPLUS_TV_Score_MMDD_Report.html`(현재 로컬에 없으면 `git log --diff-filter=A -- "LIFEPLUS_TV_Score_*.html"`로 탐색)을 템플릿으로 활용한다. 오래된 파일명(예: 0622)을 하드코딩해서 참조하지 말 것 — 리포트가 갱신될 때마다 이전 버전은 삭제되고 최신 버전만 남는 경우가 많다.

### 반드시 갱신할 항목

1. **`<title>` 태그**: `LIFEPLUS TV Score — MMDD Report`
2. **헤더 날짜/메타**: `MMDD 업데이트`, `집계완료 N편`, `신규 발행 대기 N편`
3. **DATA 배열**: JSON의 `episodes`를 LTV Score(v2) 내림차순으로 정렬하여 갱신
   - 기존 에피소드(직전 리포트에 이미 있던 편): 7D 지표는 발행 후 고정되므로 **재계산 불필요 — v2를 그대로 유지**하고 `v1`도 동일 값으로 설정 (변동 없음을 그대로 반영). 굳이 다시 계산해서 미세한 반올림 차이를 만들지 말 것
   - 신규 에피소드(이번에 처음 7D 확정된 편): 공식대로 v2 계산, `v1:null`, `isNew:true` 설정 → 순위표에 "신규" 배지 표시됨
   - `episodes` 중 이전 시즌 tail(예: 위클리 시즌 넘어가기 직전의 num이 다른 접두사인 편)은 직전 리포트에서 이미 제외됐다면 이번에도 동일하게 제외 — DATA 배열 총편수가 전주 대비 갑자기 크게 늘면 이 필터링을 빠뜨렸는지 의심할 것
4. **PENDING 배열**: JSON의 `pending` 항목으로 교체 (`status=collecting`은 1D만 있고 7D 없는 편 · `status=pending`은 미발행 편)
5. **도넛 차트 데이터**: S/A/B/C 편수를 DATA 배열에서 계산하여 하드코딩
6. **IP 평균 LTV / IP 등급분포**: DATA 배열 기반으로 동적 계산 코드 유지 (하드코딩 금지) — 이 두 차트는 이미 JS로 동적 계산되므로 DATA만 갱신하면 자동 반영됨
7. **레이더 차트**: IP별 5개 지표(N_nat 절충값·N_vr·N_ctr·N_eng·N_sub)의 평균을 DATA 배열 기반으로 재계산 — 과거 버전처럼 참여도 축을 0으로 하드코딩하지 말 것 (eng_7d가 실제로 점수에 반영되므로 레이더에도 반영돼야 함)
8. **KPI 카드**: 채널 평균 LTV(DATA의 v2 평균), S+A / B / C / 신규 편수
9. **공식 섹션 배지**: 채널 평균 LTV 업데이트
10. **상관관계 패널**: 여기 있는 피어슨 상관계수(r값)는 DATA 배열에서 실시간 계산되는 게 아니라 별도 통계 분석으로 산출된 값 — 편수가 소폭(수 편) 늘어난 정도로는 재계산하지 않고 유지해도 무방. 단, 편수가 크게 늘거나(10편+) 사용자가 명시적으로 요청하면 재계산할 것

### DATA 배열 필드 매핑 (JSON → HTML)

| HTML 필드 | JSON 필드 | 비고 |
|-----------|-----------|------|
| ip | ip | w/h/d |
| e | episode | 에피소드명 |
| v2 | 계산값 | LTV Score (기존 편은 이전 값 유지) |
| v1 | 이전 v2 | 신규 편은 null |
| nat | nat_abs_7d | HTML DATA의 `nat` 필드는 절대수(회) |
| natPct | nat_pct_7d | 소수(예 0.317) — **2026-07-08부터 필수**. 과거엔 "점수 계산 시 블렌딩만 하고 DATA엔 저장 안 함"이었는데, 그 결과 "점수 분해" 탭의 `calcComp()`가 자연유입 절대수만 단독으로 써서(절충 공식 미반영) 실제 v2와 어긋나는 분해 그래프를 그리고 있었다(2026-07-08 발견·수정). `calcComp`가 `(nm(d.natPct,b.natPct)+nm(d.nat,b.nat))/2`로 블렌딩하도록 고쳤으므로, DATA에 natPct가 없으면 그 편의 분해 막대가 다시 틀어진다. |
| vr | vr_7d | 조회율 |
| ctr | ctr_7d | CTR |
| eng | eng_7d | 참여도(좋아요+공유+댓글) 절대수, 없으면 0 |
| sub | sub_7d | 구독자증감 |
| watch | watch_min_7d | 평균시청시간(분) |
| b_org/b_algo/b_srch/pen | 계산값 | 위 "보너스/패널티 기준" 참고 |
| isNew | — | 이번에 처음 7D 확정된 편만 true |

## 3단계 — 검증 체크리스트

- [ ] 도넛 차트 S+A+B+C 합계 = DATA 배열 총 편수
- [ ] KPI 카드 편수 합계 = 집계완료 편수
- [ ] 헤더의 `집계완료 N편` = DATA 배열 실제 개수
- [ ] 채널 평균 LTV = DATA 배열 v2의 산술 평균 (소수점 2자리)
- [ ] **HTML을 저장하기 전에 `<script>` 블록만 추출해서 `node --check`로 문법 검증할 것** — 44개 이상의 DATA row를 손으로 옮기다 보면 중괄호 개수가 어긋나기 매우 쉬움 (0706에서 실제로 괄호 하나 초과로 전체 스크립트가 깨진 적 있음). Write 직후 반드시 실행.
- [ ] DATA 배열을 별도 스크립트(Node)로 로드해서 grade 집계·평균을 다시 계산해보고 KPI 카드 숫자와 일치하는지 대조 (손으로 센 숫자를 믿지 말 것)
- [ ] 기존 편들의 v2가 이전 리포트와 다르면 왜 바뀌었는지 설명 가능해야 함 (7D 지표는 고정이므로 보통 안 바뀜 — 바뀌었다면 벤치마크를 실수로 수정했는지 확인)
- [ ] 브라우저에서 정상 렌더링 확인 요청

### 사후 발견·수정 — "점수 분해" 탭 절충 공식 미반영 (2026-07-08, 별도 세션)

사용자가 "라플위클리 누적" 별도 분석(다른 프로젝트)에 같은 공식을 적용하다가, 배포된 0708 리포트의 실제 코드와 스킬 문서를 대조하는 과정에서 발견. `calcComp()`(점수 분해 탭의 상위 20편 스택 차트 계산용)가 `nm(d.nat,b.nat)` 단독으로만 자연유입을 정규화하고 있었다 — nat_pct 블렌딩이 아예 빠진 옛날 버전 그대로 남아있었던 것. **v2 점수 자체(순위·등급)는 오프라인에서 절충 공식으로 정확히 계산돼 DATA에 박혀있어 영향 없었고**, "점수 분해" 차트의 막대 비율 표시에만 국한된 문제였다.

- **원인**: 절충 공식이 처음 도입됐을 때(`acc725e`) 오프라인 계산 스크립트와 DATA 배열의 v2 값은 갱신됐지만, 브라우저에서 도는 `calcComp()` 시각화 헬퍼 함수는 고치지 않고 방치됨. DATA에 nat_pct가 아예 저장되지 않는 구조였어서 애초에 고칠 수도 없었음.
- **수정**: (1) `BENCH_W/H/D`에 `natPct` 벤치마크 값 추가. (2) DATA 배열 45개 항목 전부에 `natPct`(nat_pct_7d, 소수 4자리) 필드 추가 — data_0708.json을 ip+nat_abs_7d로 재매칭해서 복구(45/45 매칭 성공). (3) `calcComp()`를 `(nm(d.natPct,b.natPct)+nm(d.nat,b.nat))/2`로 수정. (4) 수정 후 전체 45편을 이 공식으로 역산해서 저장된 v2와 대조 — 최대 오차 0.0011(반올림 수준)로 정합성 확인.
- **교훈**: 공식을 바꿀 때 "오프라인 계산 스크립트"와 "브라우저 시각화 코드" 두 곳 다 갱신됐는지 확인할 것 — 이번 프로젝트에서 반복적으로 나온 패턴(스크립트 vs HTML 불일치, doc vs HTML 불일치)이 여기서도 똑같이 재현됨. 다음에 공식을 또 조정하게 되면 `calcComp()`도 같이 확인.

**교훈: 사용자가 말하는 "N편 업데이트"라는 진술을 그대로 믿지 말고, 반드시 ip+nat_abs_7d(또는 num) 매칭으로 신규 7D 확정편을 스크립트로 판별할 것.** 사용자가 체감하는 "업데이트"는 스프레드시트 입력 작업(신규 1D 입력, 결측치 보완 등)을 포함하는 경우가 많아 실제 신규 7D 확정편 수와 다를 수 있음.

### 고친 흔적 (커밋 순)

1. **`77fb801` Nm 버그** — PowerShell `[Math]::Max(0, double)` 호출 시 Int32 오버로드로 음수 클램핑 실패 → `[double]` 캐스팅으로 수정 (`[Math]::Max(0.0, $val)`)
2. **`f8121c3` nat_pct → nat_abs 전환** — 광고비가 많은 에피소드일수록 nat_pct가 낮아지는 편향 발견 → 자연유입 조회수 절대수(nat_abs_7d, col 12)로 정규화 기준 교체; IP별 벤치마크 재설정 (w=100,196 h=27,286 d=54,618)
3. **`acc725e` 절충 공식** — nat_abs 단독 적용 시 신규 에피소드 불리 → `(nat_pct + nat_abs) / 2` 절충 후 분포 검토로 최종 확정
4. **(미커밋, 0706) 이 문서의 eff 벤치마크 표가 실제 구현과 불일치** — 문서는 0.0074/0.0050/0.0057(비율)로 적혀 있었으나 실제 HTML은 eng_7d 절대건수를 2,500/750/1,400으로 정규화. 실제 DATA 배열을 역산해 검증 후 이 문서를 실제 구현에 맞춰 수정(위 벤치마크 표 참고). **교훈: 이 문서의 공식 설명과 실제 HTML이 다르면 HTML을 신뢰할 것 — 문서가 나중에 갱신 누락됐을 가능성이 더 높다.**
5. **(미커밋, 0706) DATA 배열 44개 row를 손으로 옮기다 닫는 괄호 1개 초과 삽입** — `node --check`로 스크립트만 추출해 문법 검증하는 습관이 없었다면 브라우저에서 차트가 통째로 안 뜨는 채로 배포될 뻔함. **교훈: DATA 배열을 수십 줄 손으로 옮겨적는 리포트는 Write 직후 반드시 `node --check`로 검증할 것.**
6. **(2026-07-14) 자연유입 가중치 5:5 → 1:9 조정, 47편 전체 재계산** — 사용자가 "광고 영향을 받는 자연유입율 비중을 줄이는 게 맞지 않나"라고 문제제기 → 실측 검증(`nat_pct_7d + ad_pct_7d = 1`, 상관계수 -1.000)으로 자연유입율이 콘텐츠 품질이 아니라 광고 집행 여부의 거울상임을 확인 → 라플위클리 누적 프로젝트에서 5:5/3:7/1:9/0:10 시뮬레이션 비교 후 1:9로 결정, 이 리포트에도 동일 적용. 변경 범위: (1) DATA 배열 47편 전부 v2 재계산 + v1도 새 v2로 리셋(수식이 바뀌었으므로 "이전 대비 변동" 비교는 이번 주기엔 의미 없음 — 신규 2편 `isNew:true`만 기존대로 v1:null 유지) (2) `calcComp()` 블렌드 비율 동일하게 수정(과거 `acc725e`→0708 사이처럼 "오프라인 계산"과 "브라우저 시각화 코드" 불일치 재발 방지) (3) 레이더차트 IP별 nNat 평균 3개 중 딥다이브만 .41→.39로 변경(위클리·HOT&NEW는 반올림상 불변) (4) 공식·정의 패널·KPI 배지·헤더 배지 텍스트 갱신. **결과: 등급분포 불변(S6 A18 B13 C10), TOP10 불변, 채널평균 0.4999→0.4970(표시상 0.50 그대로)** — 절대 등급기준(S≥0.70 등) 특성상 위클리 누적(백분위 등급)보다 훨씬 안정적으로 나타남.
